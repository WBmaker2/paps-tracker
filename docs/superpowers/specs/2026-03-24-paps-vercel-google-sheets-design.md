# PAPS 무료 배포 전환 설계

작성일: 2026-03-24

## 목적

기존 PAPS 학생 기록 시스템 MVP를 무료 호스팅 환경에서 운영할 수 있도록 전환한다.

전환 목표는 다음과 같다.

1. 배포 환경을 무료 플랜에서 지속 운영 가능해야 한다.
2. 로컬 파일 저장에 의존하지 않아야 한다.
3. 학생 기록은 실제 Google Sheets에 바로 저장되어야 한다.
4. 학생 제출은 시트 저장 성공 시에만 완료 처리되어야 한다.

## 왜 전환이 필요한가

현재 MVP는 `demo-store`와 `PAPS_STORE_PATH` 기반 파일 저장소를 사용한다.

이 구조는 Render 유료 Persistent Disk 같은 환경에서는 유지할 수 있지만, 무료 Vercel 같은 서버리스 환경에서는 영구 저장을 기대하기 어렵다. 따라서 운영 저장소를 Google Sheets로 단일화해야 한다.

## 선택한 운영 구조

### 배포

- `Vercel Hobby`

### 실제 저장소

- `Google Sheets`

### 시트 접근 방식

- `서비스 계정`으로 서버가 직접 기록

### 제출 완료 조건

- `Google Sheets 쓰기 성공 시에만 완료`

### 범위 결정

- `완전 전환`
- 운영 경로에서는 파일 기반 `demo-store`를 제거한다.
- 개발 전용 테스트 데이터 또는 시드가 필요하면 별도 fixture로 다룬다.

## 사용자 흐름

### 교사

1. Google 로그인
2. 학교/학급/학생/세션 관리 페이지 진입
3. `구글 시트 양식 생성` 링크로 템플릿 시트를 복사
4. 서비스 계정 이메일을 해당 시트에 공유
5. 시트 링크를 앱에 입력
6. 연결 검증 성공 후 운영 시작

### 학생

1. 교사가 연 세션으로 진입
2. 이름 선택
3. 기록 입력
4. 제출
5. 서버가 시트 쓰기 성공 시에만 완료 화면 표시
6. 제출 직후 자기 누적표와 간단한 종목 그래프 확인

## 서비스 계정 방식을 선택한 이유

교사 개인 OAuth 토큰으로 Google Sheets를 쓰는 방식보다 서비스 계정이 더 적합하다.

이유:

- 교사 로그인과 시트 쓰기 권한을 분리할 수 있다.
- 로그인 토큰 만료나 재동의 문제를 줄일 수 있다.
- 학생 입력 중 권한 오류가 발생할 가능성이 낮다.
- 학교 현장에서는 초기 공유 절차 1회가 운영 안정성 측면에서 더 유리하다.

교사는 결국 다음 3단계만 수행하면 된다.

1. 시트 복사
2. 서비스 계정 이메일 공유
3. 시트 링크 붙여넣기

## Google Sheets 탭 구조

### `설정`

- 학교명
- 학년도
- 템플릿 버전
- 시트 연결 상태
- 서비스 계정 공유 안내

### `학생명단`

- 학생 ID
- 학년도
- 학년
- 반
- 번호
- 이름
- 성별
- 활성 여부

### `세션`

- 세션 ID
- 세션명
- 학년도
- 학년
- 세션 유형(`official` / `practice`)
- 입력 화면 유형(`single` / `split`)
- 대상 반 목록
- 종목
- 열림/닫힘
- 생성 교사
- 생성 시각

### `기록원본`

- append-only 원본 기록
- 기록 ID
- 제출 키(clientSubmissionKey)
- 세션 ID
- 학생 ID
- 종목
- 단위
- 측정값
- 시도 순번
- 제출 시각

### `대표값`

- 세션 ID
- 학생 ID
- 종목
- 대표 시도 ID
- 대표값
- 선택 교사
- 선택 시각

### `학생요약`

- 학교 + 학년도 범위의 학생 요약
- 학생 ID
- 학년도
- 종목
- 최신 대표값
- 직전 대표값
- 최고 기록
- 변화량

### `공식평가요약`

- 학교 + 학년도 범위의 공식 요약
- 공식 세션만 별도로 정리
- 학생 ID
- 학년도
- 종목
- 대표값
- 자동 계산 등급

### `오류로그`

- 시트 연결 실패
- 서비스 계정 접근 실패
- 제출 실패
- 대표값 반영 실패

## 데이터 원칙

- 원본 기록은 항상 `기록원본`에 append-only로 남긴다.
- 교사가 선택한 대표값만 `대표값`과 요약 탭에 반영한다.
- 공식 평가 등급은 대표값 기준으로 계산한다.
- 연습 기록은 등급을 보여주지 않고 원측정값과 향상도만 제공한다.
- `학생요약`, `공식평가요약`은 `학교 + 학년도` 범위로 유지한다.
- 새 학년이 시작되면 이전 학년도 요약과 분리한다.

## API 구조

### `GET /api/teacher/bootstrap`

로드 대상:

- 학교 설정
- 연결된 시트 상태
- 학생명단
- 세션 목록
- 최근 요약 데이터

### `POST /api/google-sheet/connect`

역할:

- 시트 링크 형식 검증
- 시트 ID 추출
- 서비스 계정 접근 가능 여부 확인
- 연결 성공 시 교사 설정에 반영

### `POST /api/students`

- 학생 등록 및 수정
- 실제 저장은 `학생명단` 탭 기준

### `POST /api/sessions`

- 세션 생성
- 실제 저장은 `세션` 탭 기준

### `PATCH /api/sessions/:id`

- 세션 열림/닫힘
- 입력 화면 유형 변경

### `POST /api/sessions/:id/submit`

가장 중요한 API다.

처리 순서:

1. 세션 열림 여부 검증
2. 학생과 세션 대상 반 일치 여부 검증
3. 학생 활성 여부 검증
4. 종목과 입력값 검증
5. 시트 연결 상태 확인
6. 제출 키(`clientSubmissionKey`) 생성 또는 전달
7. 동일 제출 키가 이미 `기록원본`에 있는지 먼저 조회
8. 없을 때만 `기록원본` 탭에 append
9. 성공 시에만 완료 응답
10. 직후 조회용 학생 요약 계산

실패 시:

- 완료 처리하지 않음
- 학생에게 재시도 메시지 반환
- 가능한 경우 `오류로그`에 남김

중복 제출 방지 원칙:

- 학생 제출마다 클라이언트가 고유 제출 키를 생성한다.
- 서버는 시트 append 전에 같은 제출 키 존재 여부를 먼저 확인한다.
- Google Sheets에서는 원자적 uniqueness 보장이 어려우므로, 조회 후 append만으로 완전한 중복 방지는 보장하지 않는다.
- 따라서 읽기 계층과 요약 계층은 같은 제출 키가 여러 줄인 경우 가장 이른 한 줄만 유효 기록으로 간주한다.
- 같은 제출 키의 나머지 행은 중복 후보로 표시하고 통계 계산에서는 제외한다.
- 중복 후보는 `오류로그`와 교사 결과 화면에 노출한다.
- 네트워크 타임아웃 후 학생이 재시도하더라도 최종 요약과 대표값 계산에서는 같은 제출 키 중 하나만 반영된다.

### `POST /api/records/:id/representative`

- 교사가 대표 시도 선택
- `대표값`, `학생요약`, `공식평가요약` 반영

반영 순서:

1. `대표값` 탭 갱신
2. `학생요약` 갱신
3. 공식 세션인 경우 `공식평가요약` 갱신
4. 각 단계의 성공/실패를 `오류로그`에 기록

복구 원칙:

- 세 탭을 완전한 트랜잭션으로 묶을 수 없으므로 순차 반영으로 처리한다.
- 중간 실패가 나면 교사 화면에 `재계산 필요` 상태를 표시한다.
- 교사가 대표값을 다시 저장하거나 `요약 재계산` 작업을 실행하면 세 탭을 다시 동기화한다.

추가 복구 경로:

- `POST /api/results/rebuild`
- 학교 + 학년도 범위를 받아 `기록원본`과 `대표값` 기준으로 `학생요약`, `공식평가요약`을 다시 생성한다.
- 교사 결과 화면에는 `요약 재계산` 버튼을 둔다.

### `GET /api/results/:sessionId`

- 교사 결과 표
- 시도 목록
- 대표값 상태
- 학생별 그래프 데이터

### `POST /api/results/rebuild`

- 학교 + 학년도 범위의 요약 재생성
- `학생요약`, `공식평가요약`을 덮어써서 복구

## 오류 처리 원칙

### 학생 제출 실패

- 메시지: `지금 저장에 실패했습니다. 다시 시도해 주세요.`
- 제출 성공으로 보지 않는다.
- 다음 학생으로 자동 전환하지 않는다.

### 시트 링크 오류

- 형식이 잘못된 링크는 즉시 거절
- 서비스 계정이 공유되지 않은 시트도 즉시 거절

### 세션 닫힘

- 학생 제출 차단

### 비활성 학생

- 학생 선택 목록에서 제외
- API 제출에서도 비활성 학생이면 거절
- 과거 기록은 유지하되 새 제출만 막는다.

### 대표값 반영 실패

- 원본 기록은 보존
- 교사에게만 오류 표시

## 구현 원칙

### 제거 대상

- 운영 경로의 `demo-store` 의존
- 운영 경로의 `PAPS_STORE_PATH`
- 파일 기반 읽기/쓰기 로직

### 유지 대상

- PAPS 규칙 검증 로직
- 공식 등급 계산 로직
- 학생 입력 UX
- 교사 세션/대표값 UX

### 새로 필요한 것

- Google Sheets 읽기/쓰기 클라이언트
- 시트 행 직렬화/역직렬화 계층
- bootstrap용 시트 조회 계층
- 서비스 계정 접근 검증 계층
- 제출 키 기반 중복 방지 계층
- 대표값 요약 재계산 계층

## 기존 demo-store 데이터 마이그레이션

완전 전환 전에 기존 파일 저장 데이터가 있는 경우를 대비해야 한다.

원칙:

- 기존 `demo-store` 데이터는 자동 폐기하지 않는다.
- 관리용 일회성 마이그레이션 스크립트로 읽어서 시트 형식으로 변환한다.
- 운영 전환 직전에 `학생명단`, `세션`, `기록원본`, `대표값` 탭에 backfill 할 수 있어야 한다.
- 실제 운영 데이터가 없는 개발 환경은 빈 시트에서 바로 시작할 수 있다.

## 테스트 전략

### 단위 테스트

- 시트 링크 검증
- 서비스 계정 연결 검증
- 원본 기록 append payload 생성
- 대표값 반영 payload 생성
- 공식 등급 계산 유지

### 통합 테스트

- 학생 제출 성공 시 완료 응답
- 시트 쓰기 실패 시 미완료 응답
- 교사 bootstrap 로딩
- 세션 열림/닫힘 제어

### 배포 검증

- Vercel preview deploy 성공
- 루트 응답 확인
- 교사 로그인 리디렉션 확인
- 학생 세션 제출 흐름 확인

## 리스크와 대응

### 리스크 1. Google Sheets API 지연 또는 실패

대응:

- 학생 제출 성공 조건을 명확히 유지
- 실패 메시지를 즉시 보여주고 재시도 가능하게 한다.

### 리스크 2. 시트 구조 변경으로 인한 파손

대응:

- 템플릿 버전 필드 유지
- 탭 헤더 검증 후 연결 허용

### 리스크 3. 무료 환경의 실행 제한

대응:

- 서버 로컬 파일 의존 제거
- 시트 중심 구조로 단순화

## 결론

이 전환의 핵심은 `무료 배포`가 아니라 `영구 저장의 기준을 Google Sheets로 단일화`하는 데 있다.

이 구조로 바꾸면:

- 무료 Vercel 배포가 가능해지고
- 학생 제출 성공 조건이 분명해지며
- 학교 현장 운영 흐름도 유지할 수 있다.
