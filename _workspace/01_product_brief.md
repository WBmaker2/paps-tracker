# 제품 브리프

## 기능명

- 교사 결과 화면 필터 및 검색 패널

## 요청 요약

- 교사 결과 화면에서 특정 학생과 기록을 더 빨리 찾을 수 있도록 `학년`, `반`, `학생명`, `종목`, `세션 유형` 기준 필터와 검색 기능을 추가한다.
- 현재 결과 화면은 대표값 선택과 요약 재계산에는 강하지만, 기록이 많아질수록 원하는 학생 기록을 바로 찾기 어렵다.
- 이 기능은 교사가 수업 직후 특정 학생 기록을 검토하거나, 공식평가 대상만 빠르게 모아 확인하는 흐름을 더 짧게 만드는 데 목적이 있다.

## 왜 지금 필요한가

- 실제 운영에서는 한 학급 또는 여러 세션의 기록이 누적되면, 현재 카드형 결과 목록만으로는 원하는 학생을 빠르게 찾기 어렵다.
- `학생요약 미리보기`, `공식평가요약`, `대표값 선택` 기능은 이미 있지만, 탐색 도구가 약해 결과 검토 시간이 길어질 수 있다.
- 이 기능은 프론트엔드 UI 개선만이 아니라, 결과 데이터 선택 기준과 표시 규칙을 함께 정리하는 좋은 예시가 된다.

## 사용자

- 1차 사용자: 교사
- 사용 맥락:
  - 측정 직후 대표값을 고르는 시간
  - 공식평가 기록만 따로 확인하는 시간
  - 특정 학생의 최근 시도 기록을 다시 찾는 시간

## 사용자 목표

- 교사는 결과 화면에서 원하는 학생 또는 학급 기록을 `몇 초 안에` 찾고 싶다.
- 교사는 `공식평가만`, `연습기록만`, `특정 종목만` 빠르게 좁혀서 검토하고 싶다.
- 교사는 필터가 적용된 상태에서도 기존 `대표값 선택`, `요약 재계산`, `요약 다운로드` 흐름을 그대로 이어가고 싶다.

## 핵심 사용자 시나리오

### 시나리오 1. 학생 이름으로 빠르게 찾기

- 교사가 결과 화면에 들어간다.
- 검색창에 학생 이름 일부를 입력한다.
- 화면은 해당 학생 기록만 남기고 즉시 좁혀진다.
- 교사는 해당 학생의 시도 기록을 보고 대표값을 선택한다.

### 시나리오 2. 공식평가만 확인하기

- 교사가 `세션 유형: 공식평가` 필터를 선택한다.
- 결과 화면은 공식 세션에 해당하는 기록만 보여준다.
- 교사는 공식평가요약과 현재 표시 목록을 비교해 누락 여부를 검토한다.

### 시나리오 3. 특정 반과 종목으로 좁혀 보기

- 교사가 `5학년 1반`, `앉아윗몸앞으로굽히기`를 선택한다.
- 관련 기록만 남고, 결과가 없으면 빈 상태 안내가 나온다.
- 교사는 요약 재계산 전 대표값 선택을 마친다.

## 범위

### 포함

- 결과 화면 상단 또는 우측 패널에 필터/검색 UI 추가
- 필터 기준:
  - 학년
  - 반
  - 학생명 검색
  - 종목
  - 세션 유형(`공식`, `연습`)
- 필터 적용 결과에 맞는 목록/빈 상태 표시
- 필터 적용 상태에서 대표값 선택 동작 유지
- 필터 적용 상태에서 요약 미리보기와 다운로드 UI가 어색하지 않게 공존하도록 정리
- 테스트와 QA 시나리오 추가

### 제외

- Google Sheets 탭 구조 변경
- 새 저장소 추가
- 학생 화면 기능 변경
- 고급 권한 체계 추가
- 복합 다중 조건 저장이나 즐겨찾기 기능

## 완료 조건

- 교사 결과 화면에 필터/검색 UI가 추가된다.
- 필터 조합에 따라 결과 목록이 일관되게 좁혀진다.
- 필터 결과가 0건일 때 안내 문구와 초기화 액션이 보인다.
- 필터 상태에서도 대표값 선택이 깨지지 않는다.
- 기존 `학생요약 미리보기`, `공식평가요약 미리보기`, `CSV/XLSX 다운로드`는 그대로 동작한다.
- 관련 테스트가 추가되고 `lint`, `test`, `build`가 통과한다.
- QA 보고서에서 최소 다음이 검증된다:
  - 이름 검색
  - 공식/연습 필터
  - 반/종목 필터
  - 빈 상태
  - 대표값 선택과의 충돌 여부

## 제약

- 현재 기술 스택과 패턴을 유지한다.
  - Next.js App Router
  - React
  - 현재 `teacher/results` 페이지 구조
- 구글 시트 스키마는 바꾸지 않는다.
- 로그인/교사 인증 흐름은 건드리지 않는다.
- 가능하면 기존 `loadTeacherPageState`, `sheetTabs`, `ResultTable` 구조를 활용한다.
- 성능이 허용된다면 1차 구현은 단순하고 이해하기 쉬운 방식으로 간다.

## 설계 시 먼저 풀어야 할 질문

- 필터는 서버에서 걸지, 현재 로드된 결과를 클라이언트에서 좁힐지
- `대표값 선택` 이후 필터 상태를 어떻게 유지할지
- `현재 선택 학생 추이` 카드가 필터 결과와 어떤 관계를 가져야 할지
- 모바일에서도 필터 UI가 과하게 답답하지 않게 배치할 방법

## 리스크

- 현재 결과 페이지는 `activeSession` 중심이라, 여러 세션/조건 필터를 지원할 때 데이터 기준을 다시 정리해야 할 수 있다.
- 필터 로직이 페이지와 테이블에 분산되면 상태 관리가 복잡해질 수 있다.
- 대표값 선택 후 목록이 재정렬되거나 사라지면 교사 입장에서 혼란이 생길 수 있다.
- 요약 미리보기와 결과 목록이 다른 기준을 보여주면 화면 이해도가 떨어질 수 있다.

## 성공 판단

- 교사가 결과 화면에서 특정 학생을 찾는 시간이 체감상 확실히 줄어든다.
- 화면이 더 복잡해졌더라도, 기존 핵심 작업인 `대표값 선택`이 오히려 쉬워진다.
- 기능이 추가되어도 결과 화면의 정보 구조가 무너지지 않는다.
